# AISoul 产品需求主文档（Master v1）

> **文档定位**：单一事实来源（SSOT），汇总当前已对齐的产品范围与规则。  
> **实现与技术设计**：见 `implementation-architecture-api-db-security-v1.md`。  
> **历史细节**：原 `functional-spec-admin-public-v1.md` 为迭代过程稿，以本文为准。

### 项目定位（重要）

- **个人学习项目**：以理解架构与实现为主，**不承诺**与仓库内旧代码、旧 API、旧表结构 **兼容**。  
- **允许全量重写**：前后端可按目标设计 **整体替换**；是否在某一层保留旧代码由你自行取舍，**不作为需求约束**。  
- **部署**：上线、域名、HTTPS、CI/CD 等 **真要部署时再考虑**；本地开发优先。  
- **数据建模**：旧工程数据可不迁移；**新库新表**从零建模即可。  
- **数据不可丢失（第一原则）**：一旦上线承载业务数据，**持久化与可恢复**优先于「免费/省事」。见 **§4.1**。

## 1. 产品概述

### 1.1 目标

- 面向 **AI 行业**（可扩展多行业，**同一产品、单站点**），聚合 **第三方开放 API** 数据，提供 **趋势可视化** 与 **资源文章** 阅读。
- **摘要/文章正文**不调用大模型生成；**前台「热门」**默认由 **AI 约每三天生成一次** 快照并落库（服务端调度 + 趋势页约每 3 天自动拉取）；**灵感**仅管理端手动触发生成（可用大模型），**不对访客展示**。

### 1.2 部署与访问

| 区域 | 访问 | 说明 |
|------|------|------|
| 公开前台 | 免登录 | 趋势、资源文章、网站介绍与免责声明 |
| 管理端 | 登录 | 连接器、镜像/拉数、指标与板块、文章、热门定时快照（约每三天）、异动、灵感、消耗统计、静态页文案等 |

### 1.3 数据来源与合规原则

- 首期仅 **第三方 API**（免费公开 + 需 Key 的官方接口），经 **统一拉数出口**；**爬虫**为后续可选项。
- 展示须 **可追溯来源**；**AI 应用**类内容强制第三方来源标注；**自营工具/大模型**条目可不标第三方；第三方衍生内容须标注来源。
- **网站介绍与免责声明**独立页，法务审定免责全文；页脚链向全文。

## 2. 功能需求摘要

### 2.1 公开前台

| 模块 | 要点 |
|------|------|
| **趋势** | 时间范围（预设+自定义）、板块筛选、多类图表；默认视图使用 **当前生效的 AI 热门快照**；可「查看全部」用常规统计图表；无数据空态。 |
| **资源 · 文章** | 列表+详情；正文来自结构化/模板/截取，**非 LLM 写正文**；默认顺序来自 **热门快照**，可筛选板块/时间/类型等，「全部」下可其他排序。 |
| **热门（2.5）** | 系统 **约每三天 ≥1 次** 定时任务调用大模型，基于已入库聚合数据与文章元数据生成 **趋势推荐项 + 文章排序**，**落库**；访客只读快照，不实时调模型。 |
| **关于/免责** | 独立 URL；介绍 + 数据说明 + 免责全文 + 可选联系方式与更新日期。 |
| **首页** | 行业上下文 + 趋势/资源快捷入口 + 可选两模块热门预览（来自快照）。 |

### 2.2 管理端

| 模块 | 要点 |
|------|------|
| **连接器** | 仅 `api` 类型；密钥服务端存储；字段映射、同步频率、手动同步、测试连接。 |
| **镜像/统一出口** | 可配置；审计变更。 |
| **板块与指标** | 板块 CRUD；指标字典与图表、异动参与方式、聚合语义。 |
| **文章** | 入库与人工编辑；发布开关；类型与来源校验。 |
| **热门策略** | Cron、模型与提示版本、Top N、兜底、黑白名单、手动重跑（频控）、历史快照。 |
| **异动** | 稳健统计+合成评分+L1/L2+冷却+板块聚合；规则可配；与灵感「环境可能已变」联动（不自动改文）。 |
| **灵感** | 仅后台；选手动生成；上下文可编辑；版本历史；不对外。 |
| **LLM 消耗** | 按场景记录：`inspiration_generate`、`hot_rank_weekly`、`hot_rank_manual`；报表与预算。 |
| **静态页** | 网站介绍/免责内容编辑、版本、发布。 |
| **自营条目** | 工具/大模型/应用表单；应用强制来源。 |

### 2.3 明确不做（当前）

- 访客投稿、前台灵感、文章正文 LLM 摘要。
- 多站点部署、前台实时调大模型做热门。

## 3. 权限矩阵（需求级）

| 能力 | 访客 | 管理员（后台） |
|------|------|----------------|
| 读趋势、文章、关于/免责 | ✓ | ✓ |
| 读/写连接器、指标、文章、规则 | ✗ | ✓（按角色细分） |
| 灵感读写、手动生成 | ✗ | ✓（operator/admin） |
| 手动重跑热门 | ✗ | ✓（频控） |
| 系统定时热门任务 | — | 服务端自动；可审计 |

**角色建议**：`viewer`（只读）/`operator`（业务操作）/`admin`（含账号与安全策略）。首期可仅 `admin`。

## 4. 非功能需求（摘要）

### 4.1 数据持久化（第一原则：数据不可丢）

- **目标**：数据库文件或卷 **不因**进程重启、部署发版、主机漂移而 **静默丢失**。  
- **禁止依赖**：仅适合学习场景的 **无持久盘 Serverless + 本地 SQLite 文件**（容器/免费实例一重启库没了）。上线应使用 **带持久化卷的路径** 或 **托管数据库**。  
- **推荐**：**PostgreSQL**（托管实例：自动备份/时间点恢复，以服务商能力为准）；自管则必须 **定期逻辑备份**（`pg_dump`）+ 异地副本策略。  
- **SQLite**：仅当跑在 **明确持久化的单盘**（如 VPS 数据盘、挂载卷）且 **纳入备份** 时可接受；仍弱于 Postgres 的备份生态。  
- **备份节奏**：至少 **每日**全量或增量备份（学习项目可先从「每周备份 + 发版前手动导出」做起）；**恢复演练**偶尔做一次。  
- **事务**：关键写入使用事务；避免「半写入」状态不可恢复。

### 4.2 其他

- **性能**：公开读接口 P95 目标与前端约定（如首屏 &lt; 3s，可填数值）。
- **可用性**：热门任务失败告警；兜底策略避免前台长期空白。
- **可观测**：连接器成功率、LLM 调用、审计日志。
- **合规**：免责声明、ToS 遵从、日志脱敏。
