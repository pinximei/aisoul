# 24 Agent 多视角要求汇总

来源：24 个并行评审视角（产品、架构、前后端、数据、算法、QA、SRE、可视化、移动端、无障碍、合规、12类行业用户）。

目标：将分散意见去重后形成可执行要求，作为 `v2` 方案与任务拆解的补充依据。

---

## 1. P0 必须要求（开发前锁定）

## 1.1 产品与信息架构

- 明确产品边界：仅为公开开发者生态趋势雷达，非采购/合规/投资结论。
- 主路径必须闭环：`图表/列表 -> 趋势详情 -> 证据详情 -> 返回保持上下文`。
- P0 页面范围与里程碑、门禁分母完全一致，不允许多版本口径。
- 首页必须解释三件事：数据来源、统计口径、更新时间。

## 1.2 可用性与图表

- 每张关键图必须展示：当前值、变化率、样本量、更新时间。
- 筛选必须跨页面一致且 URL 可复现。
- 关键图元必须可下钻；下钻后返回保留筛选与时间。
- 空态/错态必须可读且可重试，不可白屏或无提示。

## 1.3 API 与数据架构

- API 契约必须单一权威并可机器校验（OpenAPI）。
- 列表接口统一分页语义（cursor/limit）与 429 重试语义。
- 追溯链必须可用：`trend_scores -> trend_features -> evidence -> normalized -> raw`。
- 对外趋势接口必须返回 `score_components`、`algo_version`、`taxonomy_version`。

## 1.4 internal 与运维安全

- internal 任务必须强鉴权（OIDC/短期令牌）+ 幂等键 + run 状态查询。
- 同幂等键必须返回同一 `run_id`（窗口期内）。
- 关键门禁必须自动化：P0=0、API 契约 100%、E2E>=95%、任务成功率>=95%。

## 1.5 合规与信任

- ToS 矩阵必须逐源可执行（URL、版本、允许/禁止、署名、留存、商用）。
- 删除/纠错必须前台可达，工单可查询、可通知、可审计。
- 强监管行业必须显式“非合规结论 + 二次验证”提示。
- 未完成主体与法域信息，不得对外发布。

## 1.6 用户审核门槛

- 24 人样本审核必须达标：总体 >=80%，任务B >=90%，任务C >=75%。
- 岗位/公司规模/决策层级必须覆盖，不得只采单一人群。

---

## 2. P1 高优先级要求（首版后紧跟）

- 增强 `/signals/new` 与后台 taxonomy/sources/compliance 页，减少人工维护。
- 完成回测体系：新旧版本双跑、排序稳定性与漂移告警。
- 补齐行业透镜（至少“场景包”映射），降低跨行业理解成本。
- 提供结构化“下一步建议模板”复制输出（可用于评审会纪要）。
- 增强移动端：任务 A/B 单手可完成、长列表性能优化。
- 增强无障碍：键盘可达、读屏语义、等价数据表。

---

## 3. P2 增长期要求

- 区域/行业深度视图（需新增数据源覆盖与偏差说明）。
- 订阅、导出、B2B API 等商业化能力。
- 更高级图分析（桑基/网络）在可读性验证后再引入。

---

## 4. 12行业补充要求（去重后）

- 电商：趋势必须映射到增长动作与实验假设，不能停留技术词。
- 金融：必须强化二次验证与可审计证据，不可暗示监管结论。
- 教育：需补教育场景映射与术语桥接，避免纯技术表达。
- 医疗：需明确不适用场景与风险分级，防止误用为临床依据。
- 制造：需披露数据覆盖偏差（公开生态 vs 工厂真实系统）。
- 媒体：证据与署名规则必须可直接用于编审发布。
- 游戏：要求版本可追溯与可复盘，避免“热词驱动”。
- 物流：强调调度建议必须有可验证动作与时效提示。
- 法律：证据固定与审计留痕需满足争议处理。
- HR：不得将趋势直接作为用工与筛选依据。
- 政务：强调透明可追溯与稳健上线，不可“带病发布”。
- 创始人：先保证最小可信闭环，再扩展功能与数据源。
